# 사용 예시

여섯 가지 시나리오. 각 예시는 사용자 입력 → 플러그인이 어떻게 반응하는지 흐름을 보여준다.

---

## [1] 새 프로젝트 골조 잡기 (기능 코드 전)

**입력:**
```
/skeleton .

서비스 프로필:
서비스_유형: REST API
주요_언어_프레임워크: FastAPI
개발_단계: 골조 잡는 중
예상_MAU: 10,000
팀_규모: 2-3명
데이터_민감도: 중간(개인정보)
업타임_요구: 99%
배포_환경: Docker
공개_여부: 인터넷 공개
```

**플러그인 동작:**
- 섹션 0(프로젝트 골조)만 적용
- 0.1 ~ 0.12 각 항목을 FastAPI/Docker 환경에 맞게 구체화
- 디렉토리 구조 추천 (피처별 vs 레이어별), 네이밍 컨벤션, Git 전략, pre-commit 자동화 제안
- 실제 파일 생성: `README.md`, `.env.example`, `pyproject.toml` (Ruff+Black), `.pre-commit-config.yaml`, 디렉토리 뼈대
- **기능 코드는 짜지 않음**

## [2] 출시 전 전체 감사

**입력:**
```
/audit ./src

서비스 프로필:
서비스_유형: REST API
주요_언어_프레임워크: NestJS + PostgreSQL + Redis
개발_단계: 출시 전
예상_MAU: 50,000
팀_규모: 5명
데이터_민감도: 중간(개인정보)
업타임_요구: 99.9%
배포_환경: k8s
공개_여부: 인터넷 공개
```

**플러그인 동작:**
- 섹션 0 ~ 10 전부, 🔴🟠🟡 등급 모두
- 항목별 결과:
  ```
  ✅ 1.1 Health endpoint — src/health.controller.ts:12
  ❌ 2.10 CORS — src/main.ts:8 (wildcard `*` 사용 + credentials true)
       수정: app.enableCors({ origin: ['https://app.example.com'], credentials: true })
  ⚠️ 5.2 DB 인덱스 — users.email 컬럼 인덱스 없음 (로그인 쿼리 핫패스)
  ⏭️ 8.4 read replica — MAU 50,000은 단일 DB로 충분
  ```
- 마지막에 ❌ 항목을 보안 > 운영 > 데이터 > 성능 순 우선순위로 재정렬

## [3] 신규 기능 추가 전 횡단 체크

**입력:**
```
새 기능 "결제" 추가 전에 ai-project-audit 기준으로 놓치기 쉬운 체크포인트만 알려줘.

[기능 명세 또는 설계 스케치]
```

**플러그인 동작:**
- 섹션 1(운영), 2(보안), 4(데이터 무결성) 중심으로 결제 기능에 적용되는 항목만 추출
- 예: 4.5 멱등성 (중복 결제 방지), 2.8 PII 마스킹 (카드번호 로그 금지), 1.10 retry+backoff (PG 일시 장애), 4.1 원자성 (결제+주문 트랜잭션)
- 구현 전에 체크포인트 리스트 → 사용자가 설계 단계에서 반영 가능

## [4] 이슈 발생 시 집중 점검

**입력:**
```
지금 "프로덕션에서 같은 주문이 두 번 결제되는 이슈" 발생했어.
ai-project-audit의 관련 섹션만 기준으로 현재 코드 점검하고 재발 방지 코드 줘.

[관련 코드]
```

**플러그인 동작:**
- 섹션 1(retry/backoff), 4(멱등성/낙관적 락/트랜잭션) 집중
- 4.5 멱등성 키 부재 → idempotency-key 헤더 도입 코드
- 1.10 retry → 결제 API에 retry 들어가있는지 확인, jitter 누락 시 추가
- Karpathy 3원칙 적용: 결제 모듈만 surgical patch, 인접 코드 손대지 않음

## [5] 특정 섹션만 집중 리팩토링

**입력:**
```
ai-project-audit 섹션 3(코드 아키텍처) 기준으로만
아래 코드 점검하고 문제 있는 부분 수정 코드 줘.

[코드]
```

**플러그인 동작:**
- 섹션 3 (SOLID + DRY/YAGNI/KISS/불변성/네이밍/함수 크기/순환 의존성) 만 적용
- 3.7 YAGNI 위반 발견 → 사용 안 되는 추상 클래스 지적
- 3.12 함수 크기 → 80줄 메서드 분리 제안
- Karpathy 2원칙: 분리 제안은 "필요하면" — 단순 케이스엔 그대로 두기 권장

## [6] 기능 1~2개 구현 후 최소치 점검

**입력:**
```
/audit .

서비스 프로필:
서비스_유형: REST API
주요_언어_프레임워크: Express + MongoDB
개발_단계: 기능 구현 중
예상_MAU: 1,000
팀_규모: 혼자
데이터_민감도: 낮음(공개)
공개_여부: 인터넷 공개
```

**플러그인 동작:**
- 섹션 0, 1, 2, 6만 점검 (스케일 전에 안 빠뜨릴 최소치)
- 🔴 필수 + 🟠 권장 등급만
- 🟡 중규모, 🔵 대규모는 ⏭️ "MAU 1,000은 적용 시기 아님"
- 결과: "지금 시점엔 이거 4개만 추가하면 충분" — 과한 인프라 제안 차단

## 플러그인 없이 그냥 던지는 법

[README.md](README.md) 통째 + 서비스 프로필 yaml 채워서 AI에 보내고:

```
README.md(ai-project-audit) 읽었어. 위 프로필로 [출시 전 전체 / 섹션 X만 / 신규 기능 전] 점검해줘.
```

만 입력하면 동일하게 작동한다. Karpathy 4원칙도 README의 Part A에 같이 들어있어서 한 번에 적용된다.
